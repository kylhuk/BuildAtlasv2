# v2 Plan: Real Builds + Real Optimization

This plan moves BuildAtlasv2 from placeholder outputs to a true end-to-end loop that generates complete Path of Exile builds and optimizes them with PoB-backed data.

Alignment baseline: `pob_build_generator_single_source_of_truth.md`.
- Single-machine PoC.
- PoB is the authoritative stat oracle.
- Schema and wire contracts evolve additively.

This refinement pass separates current repo facts from proposed additive v2 targets so the document does not mix implemented behavior with design intent.

## Navigation & Completion Semantics

- **Hierarchy**: Phase → Epic → Task → Sub-task, with each layer scoped to the previous and evidence-linked wherever possible.
- **Completion semantics**: A sub-task is done once its supporting evidence (command, query, artifact, API output) is captured; a task requires all its sub-tasks plus any documented checks; an epic only closes when every task is complete and acceptance criteria are satisfied with recorded evidence; a phase completes after all its epics close and its exit checks are satisfied.
- **Evidence expectations**: Favor concrete proofs—command logs, queries, artifacts, API responses, or documented observations—for every completion claim so traceability stays explicit.

## Execution Evidence Bundle Standard

- **Canonical bundle path pattern**: `evidence/phase_<phase_number>/epic_<epic_number>/bundle_<timestamp>/` with individual proof files rooted beneath the bundle directory.
- **Required files**: every bundle must include `commands.log`, `queries.log`, `api_samples.json`, and `artifacts_index.json`; `notes.md` is optional for explanatory context.
- **N/A evidence convention**: when a required artifact is not applicable, record `N/A` with a short rationale next to the missing entry inside the related log (e.g., `commands.log`) and flag it in `notes.md` so review knows it was deliberate.
- **Citation format**: reference evidence by phase/epic, file, and line (e.g., `Phase 2.Epic 1 commands.log#L42` or `Phase 5.Epic 3 artifacts_index.json`).

## 1) Current Gaps

- Generation still emits synthetic metrics in core paths, so optimization feedback is not fully PoB-authentic.
- Import metadata can default to `unknown` for `class`, `ascendancy`, and `main_skill`, which propagates low-quality build identity.
- Build detail surfaces do not yet expose complete chosen items, passive tree details, and gem link plans in API/UI.
- Worker contract is not yet the full, production PoB-evaluation payload surface.

Result: we cannot reliably claim that generated builds are complete, explainable, and optimized with real oracle feedback.

## 2) Target State and Definition of Done

A build is considered complete and working only when all are true:
- Identity: class, ascendancy, main skill, damage profile, patch/league context.
- Build structure: item loadout (including flasks/jewels where applicable), passive tree allocation, skill gems and link groups.
- Interop: PoB import/export material is present and valid.
- Evaluation: scenario metrics come from PoB-backed worker evaluation and are persisted for analysis.
- Transparency: UI and API expose the above clearly enough to audit why a build passes/fails.

Optimization is considered working only when:
- The loop uses PoB-backed objective signals, not placeholders.
- It produces measurable improvement versus a fixed baseline under the same budget, using predefined metrics (`full_dps`, `max_hit`, gate pass rate, and `budget_cost`).
- It records reproducible trial evidence (inputs, outputs, timing, status, and objective values).

### Product Intent Clarification

Early generated builds are expected to be weak; the product goal is not random artifacts with plausible stats. Random generation is merely a baseline that helps calibrate the loop, not the target outcome. The real objective is an ML optimizer that learns correlations across the skill tree, items, and gems grounded in PoB-backed evidence, and over time that optimizer must consistently outperform random search under the same budget.

## 3) External Benchmark (What the Ecosystem Treats as a Build)

From official and community build-sharing surfaces:
- Official PoE forums are active build-sharing channels and threads are patch/league sensitive.
- Maxroll and PoE Vault guides consistently include class/ascendancy, skill setup, passive tree progression, and gear context.
- poe.ninja acts as a build index/discovery surface where builds are compared by archetype and outcomes.
- Path of Building Community is the de facto build analysis tool and supports tree planning, item planning, skill links, and share-code import/export.
- Core game mechanics make several stats decision-critical (for example resistance caps/overcaps and attribute requirements), so these must remain queryable in the catalog (captured by explicit fields like `resist_*` + cap-state flags and `str`/`dex`/`int`).

Implication for v2: BuildAtlas must produce shareable, inspectable, version-aware artifacts, not only a score.

## 4) Contract Guardrails (Additive Only)

- PoB NDJSON/JSON-RPC: add optional fields; do not remove/rename existing keys in-place.
- ClickHouse: add columns/tables/views only; no destructive migrations.
- Filesystem artifacts: add versioned files/keys while preserving backward readability.
- FastAPI responses: new optional sections only; avoid breaking existing clients.
- Scenario/ruleset identifiers: preserve existing IDs and semantics.

Any non-additive change requires explicit approval and a migration/downgrade strategy.

### Non-goals for v2 initial execution

- No destructive schema migrations or column drops; all schema work must remain additive and backward compatible.
- No breaking API contract changes; new response sections are optional and existing clients keep their current payloads.
- No forced assumptions beyond documented evidence; every inference (identity, constraints, budgets) needs traceable validation before accepting results as authoritative.

### API/UI behavior requirements

- Stable column identifiers for each canonical field so saved views and integrations do not drift when new fields arrive.
- Server-side sort/filter support on every `sortable`/`filterable` column; numerical ranges for tiered stats and enums for qualitative flags.
- Text-searchable tags and gate-reason descriptions so users can locate builds by theme or validation outcome.
- Saved views/presets that compose common filters (e.g., "High life + crit", "Budget under 300"), with each view linked to stable filter definitions.
- Explicit presentation for unknown/`null` values rather than hiding columns when data is absent.

### ClickHouse and artifact guidance

- Fields marked `index_priority=high` should be materialized or indexed in ClickHouse tables (`builds`, `scenario_metrics`) to back fast sorting/searching; medium/low priority stats can live in JSON columns or artifacts with accompanying summary index fields.
- Keep the ClickHouse schema additive: add new columns or materialized views for new flags while retaining existing columns for backwards compatibility.
- Artifact blobs (PoB responses, manifest metadata) should store verbose structures (passive nodes, socket groups, full worker payloads) while ClickHouse stores the canonical searchable columns to avoid redundant updates.
- When a new field becomes mandatory, keep a derived fallback so older runs remain queryable, and explicitly flag missing data via the `missing-data rule` strategy.

Queryability guidance: prefer ClickHouse column/index for stats that users will filter/sort frequently (life/ES, DPS, resistances, cost), while detailed descriptions stay in artifacts referenced by `artifact_path`.

### Performance-safe sort/filter strategy

We guard catalog query performance by pairing the `index_priority` tiers with concrete runtime constraints:

- **High/medium/low index policy**: `high` columns must have ClickHouse columns/materialized views backing strong sort keys; `medium` fields can live in summary columns or flattened JSON with occasional secondary indexes; `low` fields stay in artifact JSON with only derived summaries for filtering. Explicitly mark expensive JSON sorts as `low` priority so they are only used with protective guards.
- **Required filter dimensions before expensive sorts**: queries sorting by costly metrics (e.g., DPS, total life, or derived progression flags) must include at least one high-cardinality filter such as `build_id`, `run_id`, `class`, or `budget_cost` bucket to keep ClickHouse from scanning unbounded rows.
- **Max default page/limit guidance**: API defaults to `limit=25` and never exposes unlimited cursors; any request that includes a sort on a `medium` or `low` priority column must explicitly request a `limit` ≤ 100 and feature cursor/offset pagination to avoid deep scans.
- **Pre-aggregated rollup/materialized view guidance**: heavy queries (e.g., profitability across scenarios) should target dedicated rollups or materialized views containing pre-aggregated counters, with metadata denoting the refresh cadence; ad-hoc queries can fall back to summarized ClickHouse tables that refresh nightly.
- **Fallback behavior on low-priority JSON sorts**: when the client asks to sort on a `low` priority, JSON/blob-backed field, the server returns an explanatory status, applies the sort only after fetching the limited result window (post-filter), or disables the sort by default and records the request in telemetry so we can promote fields to higher priority if needed.

### Column registry for saved views

Saved views must trust a stable column registry (`catalog_columns_v1`) so frontends and operator tooling do not break when we add new fields:

- Each registered column gets a stable `column_id` (e.g., `column_id: "total_life"`) that never changes and is referenced by saved filters, display order, and API projections.
- The registry carries a version identifier (`catalog_columns_v1`) that increments only when columns are added or compatibility aliases shift; saved-view metadata stores the registry version alongside the view definition so clients can resolve alias changes without requiring migration.
- Evolution is additive: new columns appear as optional entries with new `column_id`s, while existing `column_id`s remain available; we only introduce aliases/deprecations with a compatibility layer that maps old IDs to new ones for at least one release cycle.
- Deprecation strategy: mark a column as deprecated in the registry, add a compatibility alias pointing to the replacement `column_id`, and keep the original column readable for reads/writes while owners migrate saved views; after the alias cycle completes, we can safely retire the deprecated entry (keeping the alias in perpetuity for audit).

## 6) ML Design Details
Search space (initial):
- Discrete: class/ascendancy, main skill package, support package, key item template families.
- Structured: passive routing choices and socket/link layouts.
- Continuous/ordinal: budget caps, weighting knobs, and constraint margins.

Objective handling:
- Primary: scenario pass rate and PoB-backed core metrics.
- Secondary: cost efficiency and consistency across scenarios.
- Selection: Pareto-first, with scalar tie-breakers only after dominance filtering.

Evaluation budget policy:
- Early cheap gates first; spend full PoB evaluation budget only on survivors.
- Keep per-trial timing and cost in ClickHouse for optimizer accountability.

- Success gate: optimizer must beat the random baseline median objective (e.g., damage/pass rate) and median pass-rate without violating hard constraints for the sampled seeds.
- Repeatability: record the random baseline strategy and all optimizer seeds so follow-up runs can confirm convergence trends.
- Benchmark rigor should follow standard black-box optimization practice (fixed budget, repeated independent runs, consistent observer/logging schema) to avoid one-off lucky outcomes.
- Comparison data sources must be explicit: scenario metrics from ClickHouse (`scenario_metrics`), cost/context from build artifacts (`data/builds/<build_id>/...`), and run-level summaries (`data/runs/<run_id>/summary.json` when enabled).

Improvement threshold (initial target, tune after baseline):
- At least 15% median objective improvement versus baseline at equal evaluation budget.
- No regression in scenario pass rate.

Risks:
- PoB evaluation cost can dominate runtime; budget policies must be enforced early.
- Contract drift across worker/API/artifacts can silently corrupt optimizer signals.
- ClickHouse growth can become expensive without retention and aggregation strategy.

Assumptions:
- Single-machine resource envelope is enough for the initial optimization budget.
- Existing `make` workflows remain the default validation interface.

Open questions:
- Which passive tree representation should UI render first (node list, compact tree snapshot, or external PoB link)?
- Which scenario set becomes the canonical optimization target per league phase?
- Do we keep one global optimizer policy or allow profile-specific policies?
